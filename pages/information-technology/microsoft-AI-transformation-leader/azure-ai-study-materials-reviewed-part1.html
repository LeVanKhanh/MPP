Key Difference Between Traditional AI and Generative AI

Problem / Scenario / Need

An organization needs to understand how generative AI differs from more traditional AI approaches so it can select the correct technology for a business problem.

Traditional AI and generative AI can both use machine learning, but they are typically optimized for different outcomes.

Solutions

Best Practice: Distinguish Prediction and Classification from Content Generation

A key difference is that traditional AI usually analyzes existing data to classify, predict, detect, or recommend, while generative AI creates new content based on learned patterns and user instructions.

Traditional AI commonly performs tasks such as:

  • Predicting customer churn
  • Classifying transactions as fraudulent or legitimate
  • Detecting defects in images
  • Forecasting demand
  • Recommending products
  • Scoring risk

Generative AI commonly creates:

  • Text
  • Images
  • Audio
  • Video
  • Code
  • Summaries
  • Conversational responses

Why this is the best practice

This distinction helps organizations match the technology to the intended outcome.

Use traditional AI when the primary requirement is to:

  • Predict a value
  • Assign a category
  • Detect an anomaly
  • Rank or recommend options
  • Optimize a decision based on structured signals

Use generative AI when the primary requirement is to:

  • Produce new content
  • Transform or summarize information
  • Answer questions conversationally
  • Generate drafts, code, or designs
  • Combine instructions and context into a new response

Alternative: Describe the Difference by Model Architecture

Traditional AI often uses task-specific models trained for a narrow prediction or classification objective, while generative AI often uses large foundation models that can support many tasks through prompting.

Why select this alternative

This distinction is useful for technical audiences comparing:

  • Task-specific models
  • Foundation models
  • Training objectives
  • Prompt-based interaction
  • Fine-tuning and adaptation

Trade-offs

  • Not all traditional AI models are narrow.
  • Not all generative AI systems use very large models.
  • Architecture alone does not explain the business difference as clearly as the output type.

Alternative: Describe the Difference by Output Determinism

Traditional AI often produces a constrained result such as a class label, probability, score, or numeric forecast. Generative AI produces open-ended outputs whose wording or form can vary between requests.

Why select this alternative

This is useful when discussing:

  • Evaluation complexity
  • Reliability
  • Reproducibility
  • Governance
  • Human review requirements

Trade-offs

  • Traditional AI outputs can also vary when inputs or thresholds change.
  • Generative AI can be constrained to structured outputs.
  • This distinction is helpful but less fundamental than prediction versus content generation.

Key Distinction

AI typeTypical purposeExample output
Traditional AIPredict, classify, detect, rank, or recommendRisk score, category, forecast, anomaly flag
Generative AICreate or transform contentText, image, code, summary, conversation

Key Study Point

Traditional AI primarily analyzes existing data to make predictions or classifications, while generative AI creates new content such as text, images, audio, or code.

Improving AI Reliability Through Grounding

Problem / Scenario / Need

A generative AI model can produce answers that sound plausible but are inaccurate, outdated, incomplete, or unsupported.

This happens because the model generates responses from patterns learned during training rather than automatically verifying every statement against current or organization-specific information.

The solution needs to:

  • Improve factual accuracy
  • Use trusted and relevant information
  • Reduce hallucinations
  • Provide answers based on current business data
  • Make responses easier to verify

Solutions

Best Practice: Ground the Model with Trusted, Relevant Data

Grounding connects the AI solution to authoritative information that is relevant to the user's request.

The application retrieves appropriate information from sources such as:

  • Internal documents
  • Knowledge bases
  • Databases
  • Product documentation
  • Policies and procedures
  • Approved websites
  • Current business systems

The retrieved information is added to the model's context before it generates the response. This pattern is commonly implemented through retrieval-augmented generation (RAG).

Why this is the best practice

Grounding improves reliability because it:

  • Gives the model access to information that may not exist in its training data
  • Provides current and organization-specific context
  • Reduces the need for the model to infer or invent missing facts
  • Helps constrain answers to approved sources
  • Enables citations or source references
  • Allows knowledge to be updated without retraining the model

For example, an employee-support assistant can retrieve the latest human-resources policy before answering a leave-policy question.

Important implementation practices

A reliable grounding solution should:

  • Retrieve only relevant information
  • Prefer authoritative and current sources
  • Apply access controls before retrieving data
  • Preserve document metadata and source references
  • Instruct the model to answer only from the supplied evidence
  • Allow the model to state that the available information is insufficient
  • Evaluate retrieval quality separately from answer quality

Alternative: Fine-Tuning the Model

Fine-tuning adjusts a model using additional examples so that it better follows a desired style, format, task pattern, or domain behavior.

Why select this alternative

Fine-tuning may be useful when the primary need is to improve:

  • Response structure
  • Classification behavior
  • Specialized terminology
  • Consistency of tone
  • Repeated task performance

Trade-offs

  • Fine-tuning does not guarantee factual accuracy.
  • Knowledge embedded during fine-tuning can become outdated.
  • Updating information usually requires another training cycle.
  • It can be more expensive and operationally complex than updating a grounding source.
  • It is generally less suitable than grounding for frequently changing facts.

Fine-tuning and grounding can be combined when the system needs both specialized behavior and access to current knowledge.

Alternative: Prompt Engineering Without External Data

The application can provide clearer instructions, examples, constraints, and response formats directly in the prompt.

Why select this alternative

Prompt engineering is useful when:

  • The required knowledge is already available to the model
  • The problem is primarily about task clarity
  • The application is a prototype
  • External retrieval would add unnecessary complexity

Trade-offs

  • It cannot reliably provide information the model does not know.
  • It does not ensure that answers are current.
  • It may reduce hallucinations but cannot eliminate them.
  • Large prompts can increase token usage and cost.

Key Trade-Offs

ApproachBest suited forMain limitation
Grounding or RAGCurrent, factual, organization-specific knowledgeDepends on retrieval and source quality
Fine-tuningSpecialized behavior, format, or task consistencyKnowledge can become outdated
Prompt engineeringClearer instructions and lightweight prototypesDoes not add authoritative external knowledge

Key Study Point

Grounding improves AI reliability by supplying trusted, relevant, and current information at response time. It reduces hallucinations and enables answers to be supported by evidence, but its effectiveness depends on the quality of the source data and retrieval process.

Understanding and Applying Prompt Engineering to Improve Generative AI Answer Quality

Problem / Scenario / Need

A generative AI model may produce vague, incomplete, inconsistent, off-topic, or poorly formatted results when the user's instruction does not clearly communicate:

  • The task
  • The expected output
  • Relevant context
  • Constraints
  • Audience
  • Tone
  • Examples
  • Success criteria

Tailwind Traders has an internal policy assistant whose answers are fluent but sometimes:

  • Off-topic
  • Incomplete
  • Missing important policy rules
  • Insufficiently grounded in approved content

The organization needs a practical way to understand what prompt engineering does and use it to improve answer quality — without retraining the model or redesigning the architecture.

Solutions

Best Practice: Use Prompt Engineering to Guide the Model Toward More Relevant, Grounded, and Well-Structured Outputs

The primary purpose of prompt engineering is to design and refine the instructions, context, constraints, examples, and output format given to a generative AI model so it produces more relevant, accurate, consistent, and useful responses — without changing the model's internal parameters or weights.

A well-designed prompt specifies:

  • The model's role (typically set via a system message/system prompt)
  • The task to perform
  • The relevant context (e.g., supplied policy excerpts)
  • The expected output format
  • Constraints and boundaries (e.g., use only supplied sources; do not invent exceptions)
  • Few-shot examples of acceptable and unacceptable outputs
  • Criteria for a successful response, including how to signal missing information

Why this is the best practice

Fluent but off-topic or incomplete answers, like Tailwind Traders is seeing, usually mean the model lacks enough direction or relevant context — not that the model itself is deficient. Because prompt engineering changes only the input given to the model, it is the fastest and least expensive lever to try first: it requires no training data, no evaluation pipeline, and no hosting of a custom model deployment. A stronger prompt can:

  • Narrow the task and reduce ambiguity
  • Emphasize critical rules
  • Reduce unsupported speculation
  • Improve consistency and control tone/structure
  • Make omissions easier to detect
  • Produce answers in a predictable, checkable format

A useful structure is:

Role
Task
Relevant context
Rules and constraints
Required output format
Examples
Fallback behavior

For example, instead of asking:

Summarize this document.

A stronger prompt could be:

Summarize this policy document for senior managers in five bullet points.
Highlight risks, required actions, and effective dates.
Use only information from the supplied document.

For Tailwind Traders specifically:

You are an internal policy assistant.

Answer the employee's question using only the supplied policy excerpts.

Requirements:
- Include every rule that directly affects the answer.
- Cite the policy title and section.
- If the excerpts do not contain enough information, say so.
- Do not invent exceptions or approvals.
- Return: Answer, Applicable rules, Required actions, Sources.

Alternative: Ground the Model with External Data (RAG)

Grounding — most commonly implemented as retrieval-augmented generation (RAG) — supplies trusted, current, or organization-specific information, such as the actual policy excerpts, to the model at response time.

Why select this alternative

Choose grounding when the root cause is missing or outdated knowledge rather than instruction quality. A prompt can tell the model to "use only the supplied policy excerpts," but it cannot supply those excerpts itself — that is retrieval's job. In practice, prompt engineering and grounding are used together: retrieval supplies the facts, and the prompt tells the model how to use them.

Trade-offs

  • Requires a retrieval or data-access architecture (indexing, search, refresh pipeline)
  • Poor retrieval quality (irrelevant or outdated matches) degrades answer quality regardless of how good the prompt is
  • Adds operational and governance complexity (access control, content freshness, source attribution)

Alternative: Try a More Capable Model

Swapping to a more capable base model, without changing the prompt, is sometimes proposed as a quick fix for weak answers.

Why select this alternative

A more capable model can genuinely improve reasoning or instruction-following on hard tasks. But for an off-topic or incomplete answer, the more targeted and reversible fix is usually to improve the prompt and grounding first — including adding explicit reasoning instructions (e.g., asking the model to identify applicable rules step by step before answering) — since a stronger model given the same vague instructions and missing context will often produce the same class of error.

Trade-offs

  • Increases per-call cost and, potentially, latency
  • Does not fix poor instructions or missing context on its own
  • Should be evaluated after prompt and grounding improvements, not instead of them, so the actual cause of the quality problem is understood

Alternative: Fine-Tune the Model

Fine-tuning changes model behavior by training it further on additional labeled examples, which updates the model's parameters and typically requires deploying and hosting a distinct fine-tuned model.

Why select this alternative

Fine-tuning may be worth the added cost and complexity when the organization needs:

  • Highly consistent behavior on a narrow, well-defined task across a large number of edge cases too numerous to enumerate as few-shot examples
  • Specialized or domain-specific terminology and phrasing that examples in a prompt don't reliably teach
  • Performance improvements that prompting and grounding alone cannot achieve

Note: basic output formatting (e.g., returning JSON, or a fixed section layout) is usually solved more cheaply with prompt constraints or structured-output/JSON-mode features than with fine-tuning.

Trade-offs

  • Requires curated, labeled training data and an evaluation process
  • Costs more and takes longer than prompt refinement — in Azure OpenAI, a fine-tuned model also carries an ongoing hourly hosting cost for the deployment, on top of the one-time training cost
  • Does not guarantee access to current facts — it bakes in patterns from training data at a point in time, so it still needs grounding for anything that changes afterward
  • If the underlying base model is retired or updated, the fine-tuned model may need to be retrained against the new base

Key Distinction

ApproachPrimary purposeChanges model weights?
Prompt engineeringImprove how the model interprets instructions and structures responsesNo
Grounding / RAGProvide trusted, current, or organization-specific knowledge at response timeNo
Model upgradeIncrease raw reasoning/instruction-following capabilityNo (different model, not retrained)
Fine-tuningAdjust model behavior for specialized or repeated tasksYes

Key Study Point

The primary purpose of prompt engineering is to create clear, structured instructions — role, task, context, constraints, format, and examples — that guide a generative AI model toward more relevant, accurate, consistent, and useful outputs without retraining the model. When answers are fluent but off-topic or incomplete, first improve instructions and grounding context; reserve a model upgrade or fine-tuning for cases where prompting and retrieval genuinely cannot achieve the required behavior.

Distinguishing Prompt Engineering from Fine-Tuning

Problem / Scenario / Need

Fabrikam's executive team wants to understand prompt engineering in practical terms.

A senior manager claims:

Prompt engineering is mainly about adjusting the underlying model parameters and weights in Azure OpenAI in order to change how the model behaves.

The organization needs to determine whether this statement is correct.

Solutions

Best Practice: Treat the Statement as False

The statement is false.

Prompt engineering is the practice of designing and refining the instructions, context, constraints, examples, and output format provided to a generative AI model.

It does not change the model's internal parameters or weights.

Typical prompt-engineering techniques include:

  • Defining the model's role (typically implemented via the system message/system prompt in the chat API)
  • Giving clear task instructions
  • Supplying relevant context
  • Adding few-shot examples
  • Specifying output structure
  • Setting constraints and boundaries
  • Asking the model to cite sources or state uncertainty
  • Iteratively testing and refining the prompt

Why this is the best practice

Prompt engineering changes the input given to the model, not the model itself.

It is useful because it:

  • Improves clarity and consistency
  • Reduces ambiguity
  • Guides tone and format
  • Helps the model use supplied context effectively
  • Can improve results without retraining
  • Is faster and less expensive than changing model weights

Alternative: Fine-Tuning

Fine-tuning changes model behavior by training it on additional examples, which updates model parameters or weights.

Why select this alternative

Fine-tuning may be appropriate when the organization needs:

  • Consistent specialized behavior
  • Domain-specific terminology
  • Repeated task patterns
  • Consistent behavior across a large number of edge cases that would be impractical to enumerate as few-shot examples
  • Performance improvements that prompting alone cannot achieve

Trade-offs

  • Requires curated training data
  • Costs more than prompt refinement, and in Azure OpenAI adds an ongoing hourly hosting cost for the deployed fine-tuned model
  • Requires evaluation and lifecycle management
  • Does not automatically provide current knowledge
  • Can reinforce bias if training data is poor

Alternative: Grounding or Retrieval-Augmented Generation

Grounding supplies trusted, current, or organization-specific information at response time.

Why select this alternative

Use grounding when the main problem is missing or outdated knowledge rather than instruction quality.

Trade-offs

  • Requires retrieval architecture and source governance
  • Poor retrieval can reduce answer quality
  • Does not change the model's internal weights

Key Distinction

TechniqueWhat changes
Prompt engineeringInstructions and context sent to the model
Fine-tuningModel parameters or weights
Grounding / RAGExternal knowledge supplied at response time

Key Study Point

The statement is false. Prompt engineering changes how you instruct and contextualize the model; fine-tuning changes the model's internal parameters or weights.

Using Structured Prompt Design for a Grounded IT Support Assistant

Problem / Scenario / Need

Adventure Works is building a generative AI assistant for internal IT support.

The assistant must:

  • Use existing knowledge articles and ticket history
  • Produce accurate, context-aware answers
  • State limitations and ask clarifying questions when information is insufficient
  • Support concise or detailed response styles
  • Avoid unnecessary changes to the underlying model or infrastructure
  • Operate within clear security and governance boundaries

A solution architect proposes investing primarily in a structured prompt design that:

  • Defines the system role, guardrails, and style
  • Includes examples of acceptable and unacceptable responses
  • Instructs the model to use retrieved evidence and avoid speculation

Solutions

Best Practice: Use Structured Prompt Engineering Together with RAG and Runtime Controls

The proposed approach is appropriate, provided the assistant also has a reliable retrieval layer and supporting security controls.

Structured prompt design should define:

  • The assistant's role and approved scope (typically defined in the system message)
  • Which sources it may use
  • How to behave when evidence is incomplete
  • When to ask clarifying questions
  • When to escalate to a human
  • Required tone and response length
  • Prohibited actions and speculation
  • Expected citation or source-reference behavior
  • Output structure and fallback messages

Few-shot examples can demonstrate:

  • A well-grounded answer
  • An answer that correctly asks for clarification
  • A response that refuses to invent missing information
  • A concise answer versus a detailed answer
  • An escalation case

Why this is the best practice

Prompt engineering is well suited to these requirements because it changes the instructions and context given to the model without changing the model's internal weights.

It can improve:

  • Consistency
  • Tone control
  • Clarification behavior
  • Use of retrieved context
  • Fallback behavior
  • Response formatting
  • Resistance to unsupported speculation

This is faster and less costly than fine-tuning when the primary need is behavioral guidance rather than changing the model's learned capabilities.

However, prompt design alone does not provide enterprise knowledge or security. The complete design should include:

  • RAG or retrieval: Supplies relevant knowledge articles and approved ticket-history content
  • Prompt orchestration: Combines system instructions, user input, and retrieved evidence
  • Identity and access controls: Ensures users retrieve only content they are authorized to access
  • Content and safety controls: Detect harmful or inappropriate input and output
  • Monitoring and evaluation: Measures groundedness, accuracy, fallback quality, and escalation behavior
  • Human escalation: Handles ambiguous, sensitive, or unresolved cases

Alternative: Fine-Tune the Model

The organization could fine-tune a model using examples of IT-support responses.

Why select this alternative

Fine-tuning may be appropriate when the organization needs:

  • Highly consistent domain-specific behavior
  • Specialized terminology
  • Consistent behavior across a high volume of edge cases that would be impractical to enumerate in a prompt
  • Performance improvements that prompting cannot achieve

Trade-offs

  • It requires curated training data.
  • It changes model weights, adds lifecycle complexity, and adds an ongoing hosting cost for the fine-tuned deployment.
  • It is not the best mechanism for frequently changing knowledge articles or ticket content.
  • It still requires retrieval for current information.
  • It is excessive when structured prompts already meet the behavioral requirements.

Alternative: Rely Only on RAG

The assistant could retrieve relevant content and send it to the model with minimal instructions.

Why select this alternative

This can work for simple prototypes.

Trade-offs

  • Retrieval provides evidence but not behavioral governance.
  • The model may still speculate, use the wrong tone, or fail to clarify ambiguity.
  • Fallbacks and escalation behavior may be inconsistent.
  • Prompt structure is still needed to define how retrieved content should be used.

Alternative: Encode All Behavior in Application Code

The application can enforce tone, fallbacks, and escalation using deterministic rules outside the model.

Why select this alternative

This is useful for:

  • Hard security boundaries
  • Mandatory escalation rules
  • Validation of structured outputs
  • Actions that must be deterministic

Trade-offs

  • Rules become difficult to maintain for nuanced language behavior.
  • Tone and conversational adaptation are better handled through prompts.
  • A hybrid approach is stronger: deterministic controls for hard boundaries and prompts for flexible language behavior.

Key Distinction

CapabilityPrimary role
Structured prompt engineeringRole, tone, guardrails, clarification, fallback, and response behavior
RAGCurrent, relevant, enterprise-specific knowledge

Key Study Point

The architect's proposal is sound: structured prompt engineering is the primary way to control role, tone, guardrails, clarification, and evidence-use behavior without modifying model weights. It should be combined with RAG, access controls, safety filters, monitoring, and human escalation for a complete enterprise solution.

How RAG Works: Retriever and LLM Roles, and the Key Advantage of Grounding

Problem / Scenario / Need

A generative AI model must answer questions using current, trusted, or organization-specific information that may not be present in its original training data. Without access to relevant external knowledge, the model may:

  • Produce outdated answers
  • Invent unsupported details
  • Miss organization-specific policies or facts
  • Provide responses that are difficult to verify

Contoso is designing a retrieval-augmented generation (RAG) assistant to solve this. The architecture includes:

  • A retriever that searches enterprise content
  • A large language model (LLM) that generates the final response

Contoso needs to confirm both why RAG is used at all, and whether the following statement about how the two components divide the work is correct:

In a RAG architecture, the retriever locates and returns relevant enterprise content, while the LLM interprets the user's question and synthesizes a natural-language answer using that retrieved context.

Solutions

Best Practice: Use RAG to Ground Responses — the Retriever Finds Evidence, the LLM Synthesizes the Answer

The statement is correct, and it describes the mechanism behind RAG's key advantage: the AI system retrieves relevant information from external data sources at response time and uses that information to generate a grounded answer.

In a typical RAG pipeline:

  1. The user submits a question.
  2. The system converts the question into a search representation, such as keywords or embeddings.
  3. The retriever searches an index, vector store, database, or enterprise content source.
  4. The retriever returns the most relevant passages or records.
  5. The retrieved content is added to the prompt or model context.
  6. The LLM interprets the question together with the retrieved evidence.
  7. The LLM generates a natural-language answer.

Typical sources for the retriever include internal documents, knowledge bases, databases, product documentation, policies and procedures, search indexes, and approved websites.

Why this is the best practice

The retriever and LLM have complementary, non-overlapping responsibilities:

  • Retriever: Finds relevant evidence from external sources.
  • LLM: Understands the request, reasons over the supplied context, and generates the response.

This separation is what gives RAG its key advantage over answering from pretrained knowledge alone. Because the model does not need to rely only on knowledge learned during training, RAG:

  • Provides current and domain-specific knowledge
  • Reduces unsupported or hallucinated answers
  • Allows responses to include citations or source references
  • Updates knowledge by re-indexing content, without retraining the model
  • Keeps source data separate from the foundation model
  • Supports access controls over enterprise information

The most important advantage is that the model can answer from relevant, authoritative evidence retrieved at runtime, rather than relying only on information learned during training.

Alternative: Fine-Tune the Model

Fine-tuning adjusts model behavior using additional training examples.

Why select this alternative

Fine-tuning is useful when the main goal is to improve:

  • Response style
  • Output format
  • Specialized terminology
  • Repeated task behavior

Trade-offs

  • Fine-tuned knowledge can become outdated.
  • Updating facts requires another training cycle, not just an index refresh.
  • Fine-tuning does not automatically provide citations.
  • It is generally less suitable than RAG for frequently changing knowledge.

Alternative: Add All Reference Information Directly to the Prompt

The application can insert relevant content manually into each prompt instead of retrieving it dynamically.

Why select this alternative

This may work for:

  • Small and fixed reference sets
  • Prototypes
  • Short documents
  • Simple one-time tasks

Trade-offs

  • Long prompts increase token usage and cost.
  • The context window limits how much information can be included.
  • Irrelevant content can reduce answer quality.
  • Manual prompt assembly does not scale as the knowledge base grows.

Alternative: Let the LLM Answer Without Retrieval

The system can send the user's question directly to the LLM with no retriever in the loop.

Why select this alternative

This may be appropriate when:

  • The question depends only on general knowledge
  • The task is creative rather than factual
  • The use case is a simple prototype
  • No trusted external data source is required

Trade-offs

  • The answer may be outdated or unsupported.
  • The model may not know enterprise-specific information.
  • Hallucination risk is higher.
  • Source citations are more difficult to provide.

Alternative: Return Retrieved Documents Without LLM Synthesis

The system can display search results or matching passages directly, skipping the generation step.

Why select this alternative

This may be appropriate when:

  • Exact source content is more important than summarization
  • Users prefer to inspect the original documents
  • The system must minimize generative interpretation

Trade-offs

  • Users must read and combine the information themselves.
  • Results may be less concise or user-friendly.
  • The system provides search rather than a conversational answer.

The statement is correct: the retriever supplies relevant evidence, and the LLM uses that evidence to interpret the request and generate a natural-language answer. This retrieval-at-response-time is exactly what gives RAG its key advantage — it grounds responses in relevant, trusted, current external information without requiring the foundation model to be retrained.

Selecting the Retrieval Layer for a Contract RAG Assistant

Problem / Scenario / Need

Tailwind Traders' legal department wants to chat over thousands of contracts without placing the contract text into the model's training data.

The assistant must:

  • Retrieve relevant clauses from a large document collection
  • Reflect document updates quickly
  • Keep enterprise knowledge separate from the foundation model
  • Support grounded answers and source citations
  • Avoid fine-tuning solely to add changing contract knowledge

Solutions

Best Practice: Use Azure OpenAI with Azure AI Search

The blank should be filled with Azure AI Search.

In the RAG pipeline:

  1. Contracts are ingested, parsed, chunked, and indexed.
  2. Azure AI Search stores searchable text and vector representations.
  3. A user's question triggers keyword, vector, hybrid, semantic, or agentic retrieval.
  4. The most relevant contract passages are returned.
  5. Azure OpenAI uses those passages as grounding context to generate the answer.
  6. The application can return citations or links to the source contracts.

Why this is the best practice

Azure AI Search is designed to provide the retrieval layer for enterprise RAG solutions. It is appropriate because it:

  • Supports text, vector, hybrid, and semantic search
  • Scales to large document collections
  • Can index PDFs and extracted document content
  • Allows changed documents to be reindexed without retraining the LLM
  • Keeps authoritative knowledge outside the model's weights
  • Supports access-controlled and source-cited enterprise search patterns
  • Integrates with Azure OpenAI and Microsoft Foundry architectures

When a contract changes, the ingestion and indexing pipeline updates the searchable content. The model can then use the new clause during subsequent retrieval without a fine-tuning cycle.

Alternative: Fine-Tune the Azure OpenAI Model

The organization could train the model on contract examples or domain-specific behavior.

Why select this alternative

Fine-tuning may be useful for:

  • Consistent legal drafting style
  • Specialized terminology
  • Repeated classification or extraction behavior
  • A narrow response format

Trade-offs

  • It is not the best way to store thousands of changing contracts.
  • Knowledge can become stale.
  • Updating clauses requires additional training and evaluation.
  • Fine-tuning does not inherently provide source citations.
  • Sensitive document content becomes part of a training workflow rather than a separately governed retrieval source.

Alternative: Put Contract Text Directly into Every Prompt

The application could add selected contract text manually to the prompt.

Why select this alternative

This may work for:

  • A single contract
  • Small proof-of-concept scenarios
  • Short, fixed document sets

Trade-offs

  • Context-window limits prevent loading thousands of contracts.
  • Token costs increase.
  • Irrelevant context can reduce answer quality.
  • Manual selection does not scale.

Alternative: Use a General Database Without Search Capabilities

The organization could store contracts in blob storage, a relational database, or another repository and retrieve them with custom code.

Why select this alternative

This can preserve the source documents and may fit existing data architecture.

Trade-offs

  • The team would have to build and maintain chunking, embedding generation, indexing, ranking, and vector search logic themselves — capabilities Azure AI Search already provides out of the box, with no built-in relevance tuning, hybrid search, or semantic ranking.
  • A content repository is a place to store contracts, not a retrieval layer that can rank and surface the right clause among thousands of documents.
  • Azure AI Search can index content directly from these same repositories, so the two are normally combined rather than treated as competing choices.

Why RAG Improves an Internal HR Assistant

Problem / Scenario / Need

Fabrikam is building an internal "Ask HR" assistant with Azure OpenAI.

The assistant currently produces generic answers because it relies only on:

  • The model's pretrained knowledge
  • The user's prompt

It does not have access to Fabrikam's own HR policies, procedures, benefits documentation, or current internal guidance.

Solutions

Best Practice: Use RAG to Ground Answers in Fabrikam's Current HR Documents

The main benefit of introducing retrieval-augmented generation (RAG) is that it allows the assistant to retrieve relevant Fabrikam HR content at response time and use that information to generate organization-specific answers.

This means the assistant can:

  • Answer using Fabrikam's actual policies
  • Reflect updates when documents change
  • Reduce generic or unsupported responses
  • Provide citations or source links
  • Avoid retraining the model whenever a policy changes

Why this is the best practice

RAG adds a trusted enterprise knowledge layer without changing the model's internal weights.

A typical flow is:

  1. HR documents are indexed in a search service.
  2. The user asks a question.
  3. The retriever finds the most relevant policy passages.
  4. Those passages are added to the model context.
  5. Azure OpenAI generates a grounded answer using the retrieved evidence.

This improves reliability because the model answers from current internal sources rather than only from general pretrained knowledge.

Alternative: Fine-Tune the Model on HR Documents

The team could fine-tune a model using HR examples.

Why select this alternative

Fine-tuning may help with:

  • Tone
  • Terminology
  • Repeated response patterns
  • Consistent formatting

Trade-offs

  • It is not the best way to store frequently changing policy knowledge — benefits and procedures change on a schedule fine-tuning cycles can't keep up with.
  • New policy updates require additional training before the assistant reflects them.
  • An employee asking about a policy would get no source link back to the specific HR document, which matters when answers affect pay, leave, or compliance.
  • RAG is better for current, source-backed enterprise information.

Alternative: Add All HR Policies Directly to the Prompt

The application could insert policy text into each request.

Why select this alternative

This may work for a very small and fixed policy set.

Trade-offs

  • It does not scale to large document collections.
  • Token usage and cost increase.
  • Context-window limits apply.
  • Irrelevant content may reduce answer quality.

Key Study Point

RAG improves the HR assistant by grounding responses in Fabrikam's current, authoritative HR documents at response time, producing more accurate and organization-specific answers without retraining the model.

Using Azure AI Search to Ground Enterprise Knowledge Assistants

Problem / Scenario / Need

Fabrikam wants to build a long-lived internal "search and ask" experience over policies, contracts, and FAQs.

A solution architect states:

For long-lived enterprise knowledge scenarios, you should front Azure OpenAI with Azure AI Search so the model is grounded on indexed internal content instead of relying only on prompts.

The organization needs to determine whether this is an appropriate architecture.

Solutions

Best Practice: Treat the Statement as True

The statement is true.

For long-lived enterprise knowledge scenarios, Azure AI Search is an appropriate retrieval layer in front of Azure OpenAI. It can index and retrieve relevant internal content, which is then added to the model prompt as grounding context.

This is the retrieval-augmented generation pattern:

User question
    ↓
Azure AI Search retrieves relevant indexed content
    ↓
Retrieved content is added to the prompt
    ↓
Azure OpenAI generates a grounded response

Why this is the best practice

Azure AI Search provides capabilities that ad-hoc document chunks alone do not manage well at enterprise scale, including:

  • Persistent indexing
  • Keyword, vector, and hybrid search
  • Semantic ranking
  • Integrated vectorization
  • Document chunking and enrichment
  • Filtering by metadata
  • Security-aware retrieval patterns
  • Retrieval from large and changing document collections
  • Support for citations and source attribution in the application

This improves maintainability and retrieval relevance while reducing the need to manually assemble document context for every request. Microsoft describes Azure AI Search as a knowledge retrieval layer for RAG that grounds model responses in enterprise content.

Important Qualification: "Front Azure OpenAI" Does Not Mean Send the Entire Index

The application should retrieve only the most relevant passages for each question.

A well-designed solution should:

  • Chunk content appropriately
  • Preserve source and metadata
  • Use hybrid retrieval where useful
  • Apply filters and access controls
  • Limit retrieved context to relevant passages
  • Include source references in responses
  • Evaluate retrieval quality and groundedness
  • Re-index content when source material changes

The model should not be expected to search the index by itself unless the orchestration layer or agentic retrieval capability has been configured to do so.

Alternative: Pass Ad-Hoc Document Chunks Directly to Azure OpenAI

Why select this alternative

This can be acceptable for:

  • A quick proof of concept
  • A small, temporary document set
  • A single user-provided file
  • A narrowly scoped one-time analysis

Trade-offs

  • The application must manage chunking and selection manually.
  • Context-window limits become a problem.
  • Retrieval quality is inconsistent.
  • Updating and governing a large knowledge base becomes difficult.
  • It is not a strong architecture for a long-lived enterprise search experience.

Alternative: Fine-Tune the Model on Policies and Contracts

Why select this alternative

Fine-tuning can help change model behavior, format, or task specialization.

Trade-offs

  • Fine-tuning is not the preferred way to keep factual enterprise knowledge current.
  • Updating source content requires additional training work.
  • It does not provide direct citations to source documents.
  • RAG is usually the better approach for frequently changing policies and contracts.

The same pattern holds regardless of content mix or company framing. Whether the source content is PDFs, web pages, and database records (a company-wide "ask me anything" assistant) or proposals, knowledge-base articles, and project reports (an "ask the firm" assistant), the fill-in/true-false answer is the same: index the content in Azure AI Search, retrieve with full-text/vector/hybrid/semantic search, and pass the top passages to Azure OpenAI for a grounded, cited answer.

Reviewing a Multimodal RAG Architecture for Technical Knowledge

Problem / Scenario / Need

Adventure Works wants a technical knowledge portal for field engineers that can:

  • Search manuals, troubleshooting guides, and architecture diagrams
  • Open source documents for full detail
  • Answer conversational questions using grounded evidence
  • Index thousands of documents and images
  • Support keyword, semantic, and vector retrieval
  • Reuse the knowledge layer in agents built with Microsoft Foundry

The proposed design is:

  1. Use Azure Vision to extract OCR text, captions, and tags from diagrams and photographs.
  2. Store document content and image-derived metadata in Azure AI Search.
  3. Use Microsoft Foundry to connect models and agents to Azure AI Search and manage the conversational solution.

Solutions

Best Practice: Treat the Proposal as Appropriate and Well Aligned

The proposal is appropriate and maps each requirement to a suitable Azure capability.

Architecture layerPrimary responsibility
Azure Vision / AI enrichmentExtract searchable text and visual metadata from images
Azure AI SearchIndex, retrieve, rank, filter, and return grounded passages
Microsoft FoundryBuild, evaluate, deploy, monitor, and govern the RAG application and agents

Why this is the best practice

The architecture separates three distinct concerns:

  1. Content enrichment
    • OCR extracts text embedded in diagrams and photographs.
    • Image analysis can generate tags, captions, and other metadata.
    • The derived information becomes searchable alongside normal document text.
  2. Knowledge retrieval
    • Azure AI Search supports full-text, vector, hybrid, and semantic retrieval.
    • Search results can include source metadata and document links.
    • Relevant passages are returned to the model at query time for grounded generation.
  3. Application and agent lifecycle
    • Microsoft Foundry provides the environment for selecting models, building the conversational experience, connecting knowledge and tools, evaluating the solution, managing deployments, and monitoring it over time.
    • Foundry agents can connect to Azure AI Search indexes or knowledge bases to obtain citation-backed grounding.

Recommended End-to-End Flow

Repositories
    ↓
Document extraction + OCR/image enrichment
    ↓
Chunking, metadata, embeddings, and indexing
    ↓
Azure AI Search
    ↓
Keyword + vector/hybrid retrieval
    ↓
Microsoft Foundry model or agent
    ↓
Grounded answer with citations and source links

Important Qualification: Azure AI Search Can Orchestrate Image Enrichment

The design does not necessarily require a completely separate preprocessing application that calls Azure Vision for every image.

Azure AI Search skillsets can invoke built-in OCR and image-analysis skills during indexing. This can simplify ingestion and keep enrichment, field mapping, and indexing in one pipeline.

A separate Azure Vision preprocessing service may still be useful when:

  • The same visual metadata is needed by multiple applications
  • Custom preprocessing is required
  • Images must be analyzed before entering the search pipeline
  • The organization needs independent scaling or lifecycle control

Important Qualification: Consider Multimodal Retrieval, Not Only OCR and Tags

OCR and tags are useful, but they may not fully represent complex diagrams.

For better diagram and image retrieval, evaluate:

  • Image verbalization
  • Multimodal embeddings
  • Separate text and image vector fields
  • Retrieval of the original image or page alongside the text result

Azure AI Search supports multimodal indexing patterns using image verbalization or multimodal embeddings.

Important Qualification: "Multiple Channels" Requires a Delivery Layer

Microsoft Foundry can manage and deploy the AI application or agent, but channel delivery should be designed explicitly.

Depending on the target channel, the solution may also require:

  • A custom web or mobile front end
  • An API layer
  • Microsoft Teams or Microsoft 365 integration
  • Copilot Studio or another channel-specific integration
  • Authentication and authorization for each channel

Foundry is the application and agent platform; it does not automatically publish every solution to every possible channel without integration work.

Alternative: Store Document Chunks Directly in Prompts

Why select this alternative

This can work for a small proof of concept.

Trade-offs

  • It does not scale to thousands of documents.
  • Retrieval and updates become difficult to manage.
  • Source links, filtering, and relevance ranking are weaker.
  • Large prompts increase token cost.

Alternative: Use Azure Vision Alone

Why select this alternative

Azure Vision is appropriate for extracting text and visual information.

Trade-offs

  • It is not a persistent enterprise retrieval engine.
  • It does not replace full-text, vector, hybrid, or semantic search.
  • It does not provide the complete RAG application lifecycle.

Key Study Point

The proposal is appropriate. Use Vision-based OCR and image enrichment to make diagrams and photographs searchable, Azure AI Search as the persistent hybrid/vector retrieval layer, and Microsoft Foundry to build, evaluate, deploy, and operate the grounded assistant and agents. Consider using Azure AI Search skillsets for integrated enrichment and multimodal embeddings for complex diagrams.

Reviewing Secure RAG for a Customer-Facing Support Copilot

Problem / Scenario / Need

Adventure Works wants a customer-facing support copilot that:

  • Uses current approved documentation from multiple repositories
  • Produces grounded answers with citations or source links
  • Respects Microsoft Entra identities and document-level permissions
  • Keeps the foundation model unchanged
  • Minimizes custom model training and operational overhead

The proposed solution uses Azure OpenAI On Your Data with Azure AI Search, document security filters, and citations.

Solutions

Best Practice: Use RAG with Azure AI Search, but Do Not Start a New Design on Azure OpenAI On Your Data

The proposal is correct in its RAG architecture, but the named implementation should be updated.

The sound architectural pattern is:

  1. Index approved content in Azure AI Search.
  2. Store or ingest document-level permission metadata.
  3. Enforce security trimming at query time using Microsoft Entra identities, groups, or source ACLs.
  4. Retrieve only the passages the current user is authorized to access.
  5. Supply those passages to a model in Microsoft Foundry.
  6. Generate grounded answers with citations and links to source documents.

However, Azure OpenAI On Your Data is deprecated and scheduled for retirement on October 14, 2026. Microsoft recommends migrating these workloads to current Microsoft Foundry retrieval and agent capabilities, including Foundry Agent Service with Foundry IQ knowledge bases.

Why this is the best practice

RAG meets the requirements without changing the base model:

  • Knowledge remains in indexed enterprise content rather than model weights.
  • Updated documents can be re-indexed without fine-tuning.
  • Retrieval can be permission-aware.
  • Responses can contain source citations.
  • The model receives only the most relevant authorized passages.

This is generally more maintainable than fine-tuning for changing policies, warranties, and troubleshooting content.

Important Qualification: Microsoft Entra RBAC and Document-Level Authorization Are Different Controls

The architecture must distinguish between:

  • Service-level RBAC: controls who can manage or query the Azure AI Search service.
  • Document-level access control or security trimming: controls which indexed documents a particular end user may retrieve.

Assigning an Azure AI Search reader role does not automatically preserve the permissions of every source document.

The indexing and query pipeline must carry permission metadata and enforce it during retrieval. Azure AI Search now natively supports pushing Microsoft Entra ACL/RBAC metadata into the index (stored alongside the text and vector fields) and validates the caller's Entra token at query time to trim results automatically, including for vector search. Depending on the repository and Azure AI Search feature used, this can also involve:

  • Ingested source ACL metadata
  • User and group identifiers stored in index fields
  • Query-time security filters
  • SharePoint or Azure Storage permission-aware indexing patterns

Important Qualification: Citations Require Source Metadata and Application Support

The index should retain fields such as:

  • Source document title
  • Original URL or document location
  • Page, section, or chunk identifier
  • Repository
  • Version or effective date
  • Access-control metadata

The application or agent must then return the citation metadata with the generated answer.

Approved repositories
        ↓
Permission-aware ingestion and indexing
        ↓
Azure AI Search
  - keyword/vector/hybrid retrieval
  - document-level access enforcement
        ↓
Foundry retrieval or agent orchestration
        ↓
Azure OpenAI / Foundry model
        ↓
Grounded answer with citations and source links

Alternative: Fine-Tune the Model on Support Documents

Why select this alternative

Fine-tuning may help with response style or specialized task behavior.

Trade-offs

  • It is not the preferred way to keep changing factual knowledge current.
  • It does not inherently preserve document permissions.
  • It does not naturally provide source citations.
  • Updating knowledge requires additional training and validation.

Key Study Point

The RAG design aligns with the requirements: use Azure AI Search for permission-aware retrieval and ground a Foundry model on authorized passages with citations. However, Azure OpenAI On Your Data is deprecated, so new solutions should use current Microsoft Foundry retrieval or agent capabilities rather than adopting the retiring feature.
