Identifying Microsoft's SaaS AI Productivity Offering

Problem / Scenario / Need

An organization wants a Microsoft AI solution delivered as software as a service that improves employee productivity inside the applications people use for everyday work.

Typical tasks include:

  • Drafting and revising documents
  • Analyzing spreadsheets
  • Creating presentations
  • Summarizing email and meetings
  • Collaborating in Teams
  • Finding and using organizational information

Solutions

Best Practice: Use Microsoft 365 Copilot

Microsoft 365 Copilot is the Microsoft offering best described as a SaaS solution that boosts employee productivity with AI in everyday tools.

It integrates AI capabilities with Microsoft 365 applications such as:

  • Word
  • Excel
  • PowerPoint
  • Outlook
  • Teams
  • Loop

Why this is the best practice

Microsoft 365 Copilot is designed as an AI-powered productivity service for work. It supports users within the context of their existing Microsoft 365 applications and can use permitted organizational data through Microsoft Graph and Microsoft 365 services.

It is the best match when the primary requirement is to:

  • Improve individual and team productivity
  • Generate, summarize, analyze, and transform work content
  • Add AI assistance to familiar business applications
  • Provide a managed cloud service rather than build a custom AI system

Alternative: Microsoft 365 Copilot Chat

Microsoft 365 Copilot Chat provides a secure AI chat experience and access to agents.

Why select this alternative

It may be appropriate when users primarily need:

  • Conversational assistance
  • Web-grounded research
  • File-based analysis
  • Access to selected agents

Trade-offs

  • It is not equivalent to the full Microsoft 365 Copilot experience.
  • Full integration and work-grounded capabilities across Microsoft 365 applications can depend on licensing.
  • It is more chat-centric than application-centric.

Alternative: Microsoft Copilot Studio

Microsoft Copilot Studio is used to create and manage custom agents and copilots.

Why select this alternative

It is appropriate when an organization needs:

  • A custom conversational agent
  • Automated business processes
  • Connections to enterprise systems
  • Organization-specific knowledge and actions

Trade-offs

  • It requires solution design, configuration, testing, governance, and maintenance.
  • It is primarily a development and extensibility platform rather than an out-of-the-box employee productivity suite.
  • It may complement Microsoft 365 Copilot but does not replace its everyday productivity role.

Alternative: Microsoft Foundry

Microsoft Foundry supports the development and operation of custom enterprise AI solutions.

Why select this alternative

It is appropriate for teams building:

  • Custom generative AI applications
  • Retrieval-augmented generation solutions
  • AI agents
  • Model evaluation and governance workflows
  • AI applications requiring architectural control

Trade-offs

  • It is a platform for building AI solutions rather than a ready-to-use SaaS productivity application.
  • It requires engineering expertise and ongoing solution management.

Key Distinction

Microsoft 365 CopilotSaaS AI productivity across everyday Microsoft 365 tools
Microsoft 365 Copilot ChatSecure AI chat and agent access
Microsoft Copilot StudioBuild and manage custom agents
Microsoft FoundryDevelop and operate custom AI applications

Key Study Point

Microsoft 365 Copilot is the SaaS offering that boosts employee productivity by embedding AI into everyday tools such as Word, Excel, PowerPoint, Outlook, and Teams.

What Microsoft Copilot Studio Is and What It Enables Organizations to Do

Problem / Scenario / Need

An organization wants to create AI agents that can answer questions, use enterprise knowledge, interact with business systems, and automate tasks without building every component from scratch.

Fabrikam's CIO has heard that Copilot Studio lets organizations "build their own copilots" but needs a practical explanation of what that means — specifically, whether Copilot Studio is primarily:

  • A general employee chat product
  • A foundation-model training platform
  • An API for Microsoft 365 data
  • A platform for creating organization-specific AI agents

The organization needs capabilities for:

  • Creating custom agents
  • Adding instructions and organizational knowledge
  • Connecting to applications, APIs, and data sources
  • Automating multistep business processes
  • Publishing agents to employee or customer channels
  • Monitoring and governing agents centrally

Solutions

Best Practice: Use Microsoft Copilot Studio to Build, Extend, Deploy, and Manage AI Agents and Agent Workflows

Microsoft Copilot Studio enables organizations to create and manage custom AI agents and agent workflows through a graphical, low-code platform.

Organizations can use it to:

  • Build agents using natural language and graphical tools
  • Define agent instructions and behavior
  • Ground agents with enterprise knowledge sources
  • Add tools, connectors, APIs, and actions so agents can call business systems
  • Create agent flows and automated multistep workflows
  • Extend Microsoft 365 Copilot with organization-specific agents
  • Publish agents across supported channels
  • Monitor usage, performance, and outcomes
  • Apply security, lifecycle management, and governance controls

Why this is the best practice

Copilot Studio occupies a specific middle position, which is what answers the CIO's question directly: it is not a general chat product (that's Microsoft 365 Copilot), not a platform for training or fine-tuning foundation models, and not simply an API onto Microsoft 365 data (that's Microsoft Graph). It is the platform for turning existing business knowledge and processes into a reusable, governed AI agent without building a fully custom application for every use case.

It does this through a three-role collaboration model:

  • Business experts define knowledge, instructions, and workflows.
  • Developers add custom connectors, APIs, and advanced integrations.
  • Administrators manage environments, permissions, deployment, and governance.

This is more than a prompt editor: grounding, tools/connectors, agent flows, multi-channel publishing, monitoring, and governance are all first-class, built-in capabilities rather than things a team would otherwise have to assemble themselves.

Typical use cases include:

  • Employee self-service agents
  • Customer-support agents
  • HR and policy assistants grounded in approved documents
  • IT help-desk agents that retrieve knowledge and open service tickets
  • Sales and service automation, such as an agent that reads CRM data and updates opportunities
  • Agents that retrieve information and perform actions in business systems, such as triggering workflows from a support conversation

Alternative: Use Microsoft 365 Copilot Without Custom Agents

Microsoft 365 Copilot provides built-in AI productivity capabilities in Word, Excel, PowerPoint, Outlook, Teams, and Copilot Chat.

Why select this alternative

It is appropriate when users need standard productivity assistance without custom knowledge, integrations, or workflows.

Trade-offs

  • It provides less customization for organization-specific processes.
  • Complex actions or external integrations may require Copilot Studio.
  • It is designed primarily for everyday productivity rather than custom agent development.

Alternative: Use Microsoft Foundry for Pro-Code AI Applications and Agents

Microsoft Foundry supports developers building highly customized AI applications and agents.

Why select this alternative

It is appropriate when the solution requires:

  • Detailed model and architecture control
  • Custom code and orchestration
  • Advanced retrieval or evaluation
  • Specialized deployment and operational requirements
  • Integration into a custom application

Trade-offs

  • It requires more engineering expertise.
  • Development and maintenance effort are higher.
  • It is less accessible to business makers than Copilot Studio.

Alternative: Use Power Automate for Deterministic Workflow Automation

Power Automate can automate rule-based processes across applications and services.

Why select this alternative

It is appropriate when the process is primarily deterministic and does not require an autonomous or conversational AI agent.

Trade-offs

  • It is not primarily an agent-building platform.
  • It may be less suitable for reasoning, conversational interaction, and dynamic orchestration.
  • Copilot Studio can use agent flows and connected automation when both agent reasoning and workflow execution are needed.

Key Distinction

Microsoft offeringPrimary purpose
Microsoft Copilot StudioLow-code creation, extension, deployment, and governance of agents and agent workflows
Microsoft 365 CopilotReady-to-use AI productivity in Microsoft 365 apps and Copilot Chat
Microsoft FoundryPro-code platform for custom enterprise AI applications and agents
Power AutomateDeterministic, rule-based workflow and process automation
Microsoft GraphAPI access to Microsoft 365 data and relationships — not an agent-building platform

Key Study Point

Microsoft Copilot Studio enables organizations to build, customize, connect, deploy, and govern AI agents and agent workflows using low-code tools, enterprise knowledge, connectors, APIs, and automated actions. It is distinct from Microsoft 365 Copilot (ready-to-use productivity), Microsoft Foundry (pro-code AI development), and Microsoft Graph (a data API, not an agent platform).

Publishing Copilot Studio Agents Beyond Microsoft Teams

Problem / Scenario / Need

Fabrikam's head of support claims:

Copilot Studio agents can only be used inside Teams chats and cannot be published to other channels or into Microsoft 365 Copilot.

The organization needs to determine whether this statement accurately describes Microsoft Copilot Studio.

Solutions

Best Practice: Treat the Statement as False

The statement is false.

Microsoft Copilot Studio agents can be published to multiple supported channels, not only Microsoft Teams.

Depending on the agent experience, configuration, licensing, and channel support, publishing options can include:

  • Microsoft Teams
  • Microsoft 365 Copilot
  • Websites and web applications
  • Mobile or custom applications
  • Power Pages
  • Supported messaging and contact-center channels
  • Custom channels through available integration methods

Why this is the best practice

Copilot Studio is intended to support organization-specific agents across the places where employees or customers already work.

For Microsoft 365 scenarios, an agent can be connected to the Teams and Microsoft 365 Copilot channel, allowing users to access it from both experiences after it is published and made available to the intended audience.

For external or custom experiences, organizations can publish or integrate agents into supported websites, applications, and other channels.

The exact channels available depend on:

  • Whether the agent uses the classic or newer Copilot Studio experience
  • Authentication requirements
  • Agent capabilities and tools
  • Administrative and security configuration
  • Licensing and regional availability
  • Whether the selected channel supports the required features

Alternative: Publish Only to Teams

The organization can deliberately limit an internal agent to Microsoft Teams.

Why select this alternative

Teams-only publication may be appropriate when:

  • The target users work primarily in Teams
  • The organization wants a controlled initial rollout
  • The agent uses internal authentication and data
  • A limited pilot is preferred before broader distribution

Trade-offs

  • Users outside Teams cannot access the agent through other surfaces.
  • It reduces reach and channel flexibility.
  • It is a deployment choice, not a platform limitation.

Alternative: Publish to Microsoft 365 Copilot and Teams

The organization can make the same agent available in both Microsoft 365 Copilot and Teams.

Why select this alternative

This is the right fit for knowledge-worker agents where the goal is single sign-on convenience and minimal context switching. Employees already spend their day inside Teams and Microsoft 365 Copilot Chat for email, documents, and meetings, so surfacing the agent there means no separate app to install or learn, and the same Entra ID authentication and conditional access policies already governing Microsoft 365 apply to the agent automatically.

Trade-offs

  • Administrators must manage availability, permissions, and governance.
  • Channel-specific behavior and feature support should be tested.
  • The organization still needs to verify licensing and rollout requirements.

Key Distinction

Teams-only deploymentA deliberate channel choice
Microsoft 365 Copilot extensionMakes the custom agent available through the Microsoft 365 Copilot experience

Key Study Point

The statement is false. Copilot Studio agents can be published beyond Teams, including to Microsoft 365 Copilot and other supported channels such as websites and applications. Available channels depend on the agent experience, configuration, licensing, and feature support.

Enabling Copilot Studio Agents to Act Across Business Systems

Problem / Scenario / Need

Tailwind Traders wants a custom Copilot Studio agent that can:

  • Look up order status
  • Create cases in a service-management platform
  • Update records in Dynamics 365
  • Orchestrate several business actions from one Copilot conversation

The agent must do more than answer questions from knowledge sources; it must call external systems and execute business processes.

Solutions

Best Practice: Use Tools Backed by Connectors and Agent Flows

The blank should be filled with tools, connectors, and agent flows.

In Copilot Studio:

  • Tools define capabilities the agent can invoke.
  • Power Platform connectors provide access to Microsoft and third-party applications and APIs.
  • Agent flows orchestrate multistep business processes and can return results to the conversation.

For example, the agent could:

  1. Use a Dynamics 365 connector to retrieve an order.
  2. Ask the user for missing information.
  3. Invoke an agent flow to create a service case.
  4. Update the related customer or order record.
  5. Return the case number and status to the user.

Why this is the best practice

Tools are the agent-facing building blocks that enable action. Connectors provide standardized communication with external services, while agent flows coordinate sequences of actions, conditions, and business rules.

This approach supports:

  • Reusable integrations
  • Low-code orchestration
  • Authentication and connection management
  • Validation and branching logic
  • Calls to Dynamics 365 and other enterprise systems
  • A single conversational experience for multiple backend actions

Alternative: Use Knowledge Sources Only

The agent can ground answers in Dataverse, SharePoint, websites, or other knowledge sources.

Why select this alternative

This is appropriate when the agent only needs to retrieve and explain information.

Trade-offs

  • Knowledge sources do not by themselves create cases or update records.
  • Retrieval provides information, while tools and flows perform actions.
  • A transactional agent generally needs both knowledge and action capabilities.

Alternative: Use a Custom API Tool

The organization can expose proprietary services through a custom connector, REST API, or other supported tool integration.

Why select this alternative

This is useful when:

  • No prebuilt connector exists
  • The business operation requires custom logic
  • An internal API already provides the needed transaction

Trade-offs

  • More development, security testing, and maintenance are required.
  • Authentication, error handling, retries, and authorization must be designed carefully.
  • Prebuilt connectors and agent flows are usually faster for standard Dynamics 365 and service-platform integrations.

Alternative: Put All Process Logic in Topics

Topics can control conversational paths and collect information from users.

Why select this alternative

Topics are useful for deterministic dialogue, clarification, and routing.

Trade-offs

  • Topics alone do not provide backend integration.
  • Large process logic can become difficult to maintain.
  • Use topics for conversation control and tools or flows for external actions.

Key Distinction

CapabilityPrimary role
Knowledge sourceSupplies information for grounded answers
ToolGives the agent a callable capability
ConnectorConnects the agent or flow to an application or API
Agent flowOrchestrates multistep actions and business logic
TopicControls conversational paths and clarification

Key Study Point

To call external systems and orchestrate end-to-end business processes, a Copilot Studio agent should primarily use tools backed by Power Platform connectors and agent flows.

Reviewing a Copilot Studio Design for Voice Contact Center and Microsoft 365 Copilot

Problem / Scenario / Need

Adventure Works wants a conversational support solution that can:

  • Understand natural-language voice and chat
  • Support telephony and DTMF keypad input
  • Retrieve data from Microsoft 365, Dynamics 365, and external APIs
  • Use generative AI rather than only fixed IVR menus
  • Escalate complex calls to human representatives
  • Provide an internal version through Microsoft 365 Copilot

The architecture team proposes designing one Copilot Studio agent, enabling voice and telephony, adding connectors and actions, and publishing that same agent to both the contact-center channel and Microsoft 365 Copilot.

Solutions

Best Practice: Treat the Proposal as Directionally Correct but Not Fully Correct as Written

The proposal makes appropriate use of several Copilot Studio capabilities:

  • Topics and generative orchestration for conversational behavior
  • Voice integration with Dynamics 365 Contact Center
  • Speech and DTMF input
  • Call transfer or escalation to human representatives
  • Connectors, REST APIs, and tools for backend actions
  • Publication to Teams and Microsoft 365 Copilot for supported text-based agents

However, the assumption that one voice-enabled agent can simply be published unchanged as a Microsoft 365 Copilot chat agent requires correction.

Microsoft documentation distinguishes:

  • Text modality for chat-based agents
  • Speech and DTMF modality for voice-enabled agents

For basic voice agents, these modalities are mutually exclusive. Therefore, Adventure Works should normally design separate channel-specific agent experiences, or separate agents that reuse shared knowledge, tools, connectors, flows, and business logic.

Why this is the best practice

Voice and chat require different interaction design.

A contact-center voice agent must account for:

  • Speech recognition and synthesized speech
  • Silence and no-input handling
  • Barge-in behavior
  • DTMF mappings
  • Telephony latency and audio quality
  • Call context and transfer to a human
  • Short prompts that are easy to understand by listening

A Microsoft 365 Copilot agent typically uses:

  • Text-based interaction
  • Rich links and formatted responses
  • Microsoft 365 grounding and permissions
  • Longer responses that users can read
  • Employee-oriented actions and authentication

Reusing the same enterprise integrations is sensible, but the interaction layer should be optimized and tested separately for each modality.

Recommended Architecture

  1. Build a voice/contact-center agent
    • Use Copilot Studio voice capabilities.
    • Integrate it with Dynamics 365 Contact Center.
    • Configure speech, DTMF, no-input handling, and call transfer.
    • Use tools and connectors to retrieve account information and perform actions.
  2. Build a text-based internal agent
    • Publish it through the Teams and Microsoft 365 Copilot channel.
    • Reuse approved knowledge, connectors, agent flows, and APIs.
    • Adapt instructions and response formatting for employee chat.
  3. Share backend capabilities
    • Put reusable business operations in connectors, REST APIs, agent flows, or other governed tools.
    • Apply common authentication, authorization, auditing, and DLP policies.
    • Avoid duplicating core transaction logic inside channel-specific conversations.
  4. Test each channel separately
    • Validate voice accuracy, DTMF, latency, and human handoff.
    • Validate text responses, citations, permissions, and Microsoft 365 availability.

Alternative: Use One Basic Voice Agent for the Contact Center Only

Why select this alternative

This directly meets the external, customer-facing voice requirement first and reduces initial implementation scope: Adventure Works can deliver the harder, more novel piece — telephony, speech recognition, DTMF, and human handoff — as a single focused project before layering on the internal Microsoft 365 Copilot experience.

Trade-offs

  • It does not provide the required Microsoft 365 Copilot experience.
  • A separate internal agent would need to be added later.

Alternative: Use One Text Agent in Microsoft 365 Copilot Only

Why select this alternative

This is appropriate as a fast first release when only the internal, text-based use case is urgent. It reuses the same knowledge sources, connectors, and agent flows planned for the full solution, letting Adventure Works validate grounding and backend actions with employees quickly while deferring the more complex voice/telephony build.

Trade-offs

  • It does not supply PSTN telephony, speech, DTMF, or contact-center handoff.
  • It cannot replace the required voice-channel design.

Key Distinction

CapabilityAppropriate design
Natural conversational voiceCopilot Studio voice-enabled agent
PSTN telephony and DTMFDynamics 365 Contact Center integration and voice configuration
Backend system actionsTools, Power Platform connectors, REST APIs, and agent flows
Human escalationContact-center transfer and routing configuration
Microsoft 365 Copilot experienceText-based agent published to Teams and Microsoft 365 Copilot

Key Study Point

The proposal is directionally correct but needs an architectural adjustment. Copilot Studio supports voice, DTMF, telephony integration, generative conversations, backend actions, human escalation, and Microsoft 365 Copilot publishing. However, a basic voice-enabled agent uses the Speech and DTMF modality, while a chat agent uses the Text modality; these are mutually exclusive. Use separate channel-optimized agents or experiences while reusing common knowledge, tools, connectors, and flows.

Selecting a Copilot Solution for First Drafts of Policy Documents

Problem / Scenario / Need

An organization needs to create an initial draft of a policy document based on instructions, existing organizational files, or reference material.

The solution should:

  • Generate structured long-form text
  • Work in a familiar document-authoring environment
  • Use existing files as grounding material when appropriate
  • Support human review, editing, formatting, and approval

Solutions

Best Practice: Use Microsoft 365 Copilot in Word

Microsoft 365 Copilot in Word is best suited for generating first drafts of policy documents.

Copilot in Word can create new content from a prompt and can use existing documents as reference sources. The user can then review, revise, format, and collaborate on the draft directly in Word.

Why this is the best practice

Microsoft 365 Copilot in Word is preferred because it:

  • Is designed for drafting and refining long-form documents
  • Works directly within Microsoft Word
  • Can reference organizational files available through Microsoft 365
  • Supports rewriting, expansion, summarization, and tone adjustment
  • Fits existing review and collaboration workflows
  • Keeps a human author responsible for validating and approving the final policy

A suitable workflow is:

  1. Provide the policy purpose, audience, scope, and required sections.
  2. Reference approved policies, standards, or templates.
  3. Ask Copilot in Word to generate the first draft.
  4. Review the draft with legal, compliance, security, and business owners.
  5. Revise and approve the policy through the organization's normal governance process.

Copilot should be treated as a drafting assistant, not as the final authority on legal or organizational policy.

Alternative: Use Microsoft 365 Copilot Chat

Microsoft 365 Copilot Chat can generate an outline or initial policy text in a conversational interface.

Why select this alternative

It may be useful when:

  • The user is brainstorming policy content
  • The document structure has not yet been decided
  • The user wants to compare several approaches before creating the Word document

Trade-offs

  • The output may require additional transfer and formatting in Word.
  • The workflow is less document-centric than drafting directly in Word.
  • Users must still verify that referenced information is authoritative and current.

Alternative: Build a Policy-Drafting Agent with Microsoft Copilot Studio

A specialized agent can be built to retrieve approved policy sources, apply organizational templates, and guide users through a standardized drafting process.

Why select this alternative

This approach may be appropriate when:

  • Policy drafting is a frequent, repeatable enterprise process
  • The organization requires consistent templates and mandatory sections
  • Drafts must use controlled knowledge sources
  • The workflow requires approvals, integrations, or automated document generation

Trade-offs

  • It requires design, implementation, testing, security, and maintenance.
  • Governance is needed for the agent's knowledge sources and actions.
  • It is excessive for occasional, straightforward document drafting.
  • Microsoft 365 Copilot in Word remains simpler for individual first drafts.

Copilot solutionBest suited for
Microsoft 365 Copilot in WordDrafting and refining policy documents directly in Word
Microsoft 365 Copilot ChatBrainstorming, outlining, and exploratory drafting
Microsoft Copilot StudioBuilding a repeatable, governed policy-drafting workflow or custom agent

Key Study Point

Microsoft 365 Copilot in Word is best suited for generating first drafts of policy documents because it is designed for long-form document creation, can use existing files as references, and supports direct human review and refinement in Word.

Requirements for Accessing Microsoft 365 Copilot Features

Problem / Scenario / Need

An organization wants users to access Microsoft 365 Copilot capabilities in applications such as Word, Excel, PowerPoint, Outlook, and Teams.

The organization needs to understand the licensing and technical prerequisites required before Copilot features become available to each user.

Solutions

Best Practice: Assign an Eligible Microsoft 365 Base License and a Microsoft 365 Copilot License

For full Microsoft 365 Copilot features, each user generally needs:

  • An eligible Microsoft 365 or Office 365 base subscription
  • A Microsoft 365 Copilot add-on license assigned to that user
  • A Microsoft Entra ID account
  • Supported Microsoft 365 applications, services, update channels, and network access

For applicable Outlook scenarios, Microsoft also requires a supported primary mailbox hosted in Exchange Online.

Why this is the best practice

Microsoft 365 Copilot is licensed per user. Purchasing licenses at the tenant level is not sufficient by itself; administrators must assign the required licenses to the specific users who need access.

The eligible base subscription provides the underlying Microsoft 365 services, while the Copilot license enables the full Copilot experience across supported Microsoft 365 applications.

Administrators should also verify that:

  • The user's Microsoft 365 Apps are on a supported update channel
  • Required connected experiences are enabled
  • Network endpoints are not blocked
  • The user has access to the Microsoft 365 data that Copilot is expected to use
  • Existing permissions and information-protection policies are correctly configured

Copilot respects the user's existing Microsoft 365 permissions. A Copilot license does not grant the user access to documents, email, sites, or other information that the user could not already access.

Alternative: Use Microsoft 365 Copilot Chat with an Eligible Base License

Users with an eligible Microsoft 365 subscription may have access to Microsoft 365 Copilot Chat without the full Microsoft 365 Copilot add-on.

Why select this alternative

Copilot Chat can be appropriate when users need:

  • Secure, work-oriented AI chat
  • Web-grounded assistance
  • Limited agent access
  • A lower-cost entry point before assigning full Copilot licenses

Trade-offs

  • It does not provide the same complete, work-grounded Copilot experience across Word, Excel, PowerPoint, Outlook, and Teams.
  • Access to organizational data and agents can depend on licensing and consumption configuration.
  • Some advanced capabilities require the Microsoft 365 Copilot add-on.

Alternative: Enable Usage-Based Agent Access

An organization may configure metered or pay-as-you-go access for certain agents without assigning every user a full Microsoft 365 Copilot license.

Why select this alternative

This may be suitable when:

  • Users need access to a specific agent rather than the full Copilot suite
  • Usage is limited or unpredictable
  • The organization wants to pilot agent scenarios before broader licensing

Trade-offs

  • Capabilities may be more limited than those available to fully licensed users.
  • Usage costs must be monitored.
  • Tenant billing, security, and governance configuration are required.
  • It does not replace a full Copilot license for users who need Copilot throughout Microsoft 365 applications.

Distinguishing Microsoft Copilot Products and Extensibility Capabilities

Problem / Scenario / Need

An executive claims that Microsoft 365 Copilot, Microsoft Copilot, Microsoft Copilot Studio, and Microsoft Graph are interchangeable because they all contribute to AI-enabled work.

The executive therefore proposes that exam preparation should focus only on generic prompting rather than understanding which Microsoft capability fits each business process or extensibility scenario.

Solutions

Best Practice: Treat the Statement as False and Learn the Distinct Role of Each Capability

The statement is false.

These capabilities are related, but they are not interchangeable products. They serve different roles in the Microsoft AI ecosystem, and selecting the appropriate capability is a core business and architecture decision.

Microsoft 365 Copilot

Microsoft 365 Copilot is an AI-powered productivity solution integrated with Microsoft 365 experiences such as Word, Excel, PowerPoint, Outlook, Teams, and Copilot Chat.

Use it when the primary need is to:

  • Improve employee productivity
  • Draft, summarize, analyze, and transform work content
  • Use permitted organizational context from Microsoft 365
  • Provide ready-to-use AI capabilities within everyday work applications

Microsoft Copilot / Microsoft 365 Copilot Chat

The term Microsoft Copilot can refer broadly to Microsoft's family of Copilot experiences. In current workplace guidance, Microsoft 365 Copilot Chat is the secure chat-centered experience for asking questions, analyzing content, performing research, and accessing agents.

Use it when the primary need is:

  • Conversational AI assistance
  • Web-grounded or work-grounded chat, depending on licensing and context
  • Entry-point access to agents
  • General research and productivity support

Because product naming has evolved, exam preparation should pay attention to the specific product name and scenario rather than assuming every reference to "Copilot" means the same capability.

Microsoft Copilot Studio

Microsoft Copilot Studio is a graphical, low-code platform for building, extending, deploying, and governing agents and agent flows.

Use it when the organization needs to:

  • Build a custom business agent
  • Ground an agent with enterprise knowledge
  • Connect to APIs, connectors, and business systems
  • Automate business actions and workflows
  • Extend Microsoft 365 Copilot with specialized capabilities

Microsoft Graph

Microsoft Graph is a developer platform and API surface for accessing Microsoft 365 data and relationships, subject to identity, permissions, and security controls.

It is not an end-user Copilot product.

Use it when an application or agent needs programmatic access to resources such as:

  • Users and groups
  • Mail, calendars, and meetings
  • Files and SharePoint content
  • Teams and chats
  • Organizational relationships and activity context

Microsoft 365 Copilot itself uses Microsoft Graph to access organizational context that the user is permitted to access.

Why this is the best practice

Understanding these distinctions helps teams:

  • Select the correct product for the business requirement
  • Avoid unnecessary custom development
  • Recognize when extensibility or integration is required
  • Understand licensing and governance implications
  • Match SaaS productivity, low-code agents, and API-based development to the appropriate scenario
  • Answer scenario-based exam questions correctly

Generic prompting is useful, but it does not replace product and scenario knowledge.

A strong exam-preparation approach should cover both:

  1. How to communicate effectively with generative AI
  2. How to choose the appropriate Microsoft capability for a business process, data-access pattern, or extensibility requirement

Alternative: Focus Only on Prompt Engineering

The learner can study prompt structure, context, constraints, examples, and output formatting.

Why select this alternative

Prompt engineering is useful across many generative AI tools and can improve output quality.

Trade-offs

  • It does not explain which Microsoft product should be selected.
  • It does not cover licensing, data boundaries, extensibility, or governance.
  • It is insufficient for scenario-based questions about build, buy, extend, integrate, or automate decisions.

Alternative: Memorize Product Definitions Without Business Scenarios

The learner can memorize a short definition for each product.

Why select this alternative

This may help with direct recall questions.

Trade-offs

Distinguishing the Microsoft Copilot Surfaces: M365 Copilot, Copilot Chat, Copilot Pro, Copilot in Windows, and GitHub Copilot

Problem / Scenario / Need

An organization has developers working in GitHub and Visual Studio, knowledge workers across Microsoft 365, and executives who mostly use a browser and mobile device. The CIO proposes "one Copilot for everyone" to simplify licensing and training.

A related, narrower confusion shows up inside Microsoft 365 itself: a program manager assumes that once people learn to write good prompts in one app, those prompts will behave identically in every other app — and cannot explain why colleagues report that "Copilot acts differently" in Word versus Excel versus Outlook versus Teams. People are also seeing several different things labeled "Copilot" — a taskbar icon in Windows, an app on their phone, a paid personal subscription a colleague mentions, and a chat panel inside GitHub — and assume these are all the same product with different skins.

The organization needs a clear, correct picture of how many distinct "Copilot" surfaces actually exist, who each one is for, and why in-app behavior legitimately differs even within Microsoft 365.

Solutions

Best Practice: Recognize That "Copilot" Names a Family of Distinct, Separately Licensed and Governed Experiences, Not One Product

Microsoft's Copilot portfolio spans several genuinely different products, each scoped to a different persona and grounded in different data:

  • Microsoft 365 Copilot (in-app): Copilot embedded inside Word, Excel, PowerPoint, Outlook, and Teams. Requires an eligible Microsoft 365 base license plus the Microsoft 365 Copilot add-on, and is governed under Microsoft 365's enterprise data protection, tenant isolation, and compliance boundary.
  • Microsoft 365 Copilot Chat: the cross-device conversational assistant (web, desktop, iOS, Android) described elsewhere in this document — broadly available with a qualifying work/school account, with a fuller work-grounded experience for users holding the Copilot add-on.
  • Copilot Pro: a paid consumer subscription tied to a personal Microsoft account. It gives an individual priority access and Copilot features inside personal Office apps and other consumer experiences. It is not an enterprise license, is not assigned or governed through Microsoft Entra ID or the Microsoft 365 admin center, and does not carry Microsoft 365 Copilot's enterprise-grade contractual, tenant-isolation, or compliance commitments. It should not be used as a substitute for a proper Microsoft 365 Copilot or Copilot Chat license when the work involves organizational data.
  • Copilot in Windows: an OS-level assistant pinned to the Windows taskbar, providing general assistance, some system actions (opening apps, changing settings), and web-grounded question answering. It is a Windows feature, not a Microsoft 365 application experience — its scope, sign-in behavior, and management (via Windows/Intune policy) are distinct from Microsoft 365 Copilot's application-level licensing and Microsoft Graph grounding.
  • GitHub Copilot: a separate, developer-focused AI pair-programmer integrated into IDEs (Visual Studio, VS Code, JetBrains) and GitHub itself, used for code completion, in-editor chat, and pull-request summaries. It is grounded in source code and repository context — not Microsoft 365 content — and is licensed, administered, and billed entirely separately from Microsoft 365 Copilot, through GitHub/organization settings rather than the Microsoft 365 admin center.
  • Microsoft Copilot Studio: the low-code platform for building custom agents (covered earlier in this Part) — a development platform, not an end-user chat surface.

Why this is the best practice

"One Copilot for everyone" does not match how Microsoft has actually designed and licensed these products, for a concrete reason in each case:

  • A developer's needs (code completion, repository-grounded chat, PR summarization) have nothing to do with Word/Excel/Outlook grounding — GitHub Copilot exists precisely because a coding assistant needs different grounding, different IDE integration, and a different distribution channel (GitHub/IDE marketplaces) than an M365 productivity assistant.
  • A personal Copilot Pro subscription is billed to and controlled by an individual, not the organization — it sits outside Entra ID, Conditional Access, DLP, and the Microsoft 365 compliance boundary, which is exactly the wrong place for business data to end up (the same risk pattern described in the Enterprise Data Protection topic in Part 4, where an employee reaches for a consumer-facing tool because it's immediately available with no IT provisioning).
  • Copilot in Windows operates at the operating-system layer, available regardless of which (if any) Microsoft 365 apps are installed — it is scoped and administered differently from an in-app Microsoft 365 Copilot license.
  • Even within Microsoft 365 Copilot itself, each application's Copilot is grounded in and scoped to that application's own data model and available actions, which is why the same prompt does not behave identically everywhere:
    • Copilot in Word is tuned for long-form document drafting, rewriting, and summarizing, grounded in the current document and referenced files.
    • Copilot in Excel is grounded in tabular data and formulas — it can generate and explain formulas, identify trends, and create visuals from a specific range, tasks that have no equivalent in Word.
    • Copilot in PowerPoint converts source documents or prompts into slides, speaker notes, and design suggestions.
    • Copilot in Outlook drafts, summarizes, and triages email threads, grounded in the current message/thread and calendar context.
    • Copilot in Teams summarizes meetings and chats and can surface action items from a specific conversation.
    A prompt like "summarize this" therefore succeeds differently in each app because each app supplies different grounding context and exposes different underlying actions to the model — the apps are not simply different skins over one identical assistant.

This means a rollout plan should treat "Copilot" as a portfolio decision — matching each persona (developer, knowledge worker, executive/mobile user, casual OS-level user) to the specific surface built for them — rather than a single licensing and training exercise.

Alternative: Standardize on a Single Copilot License for Every Persona

The organization could attempt to give every user (developers included) the same Microsoft 365 Copilot license and train everyone on identical prompts.

Why select this alternative

It appears to simplify procurement, training materials, and support.

Trade-offs

  • Developers get no code-completion or repository-grounded capability from a Microsoft 365 Copilot license — they need GitHub Copilot specifically.
  • Training materials that assume identical in-app behavior will be wrong the moment users move from Word to Excel to Teams.
  • It ignores that Copilot Pro, Copilot in Windows, and GitHub Copilot are licensed, billed, and administered through entirely different systems than Microsoft 365 Copilot, so "one license" cannot actually cover them.

Alternative: Allow Individuals to Self-Purchase Copilot Pro for Work Use

Employees who want Copilot sooner than IT can provision it could be allowed to expense or use a personal Copilot Pro subscription.

Why select this alternative

It can appear to close a short-term gap while enterprise licensing is being rolled out.

Trade-offs

  • Work content processed under a personal account sits outside Entra ID, Conditional Access, DLP, and Purview visibility.
  • It creates exactly the "shadow AI" and inconsistent data-protection risk described in the enterprise data protection and integrated-governance topics elsewhere in this document.
  • It does not scale into a supportable, audited enterprise deployment.

Selecting a Copilot Experience for Executive Market Research

Problem / Scenario / Need

A strategy team needs to create a long-form executive briefing about a new market segment.

The briefing must:

  • Combine analyst information from the web
  • Use internal slide decks and meeting notes
  • Produce a structured, shareable report
  • Include sources or citations
  • Support content that can be transferred into PowerPoint
  • Avoid unnecessary code execution or deep statistical modeling

Solutions

Best Practice: Start with Researcher in Microsoft 365 Copilot

The team should rely first on Researcher in Microsoft 365 Copilot.

Researcher is designed for complex, multistep research that combines information from:

  • The web
  • Files and documents
  • Email
  • Meetings
  • Chats
  • Other work content the user is permitted to access

It produces structured, source-cited reports suitable for review and executive communication.

Why this is the best practice

Researcher is the strongest fit because the requirement is primarily:

  • Broad, defensible research
  • Synthesis across internal and external sources
  • Long-form report generation
  • Clear structure and source attribution

It is specifically intended for deeper research tasks that require more than a quick answer or short summary.

A recommended workflow is:

  1. Ask Researcher to investigate the market segment.
  2. Identify the relevant internal decks and meeting notes.
  3. Specify the executive audience, report structure, and decision questions.
  4. Review the cited sources and challenge unsupported conclusions.
  5. Transfer the approved report or outline into PowerPoint.
  6. Use Copilot in PowerPoint to convert the material into an executive presentation.

Alternative: Use Microsoft 365 Copilot Chat

Copilot Chat can answer questions, summarize files, brainstorm ideas, and support conversational research.

Why select this alternative

It is useful when the team needs:

  • A quick answer
  • A short summary
  • Early brainstorming
  • Iterative discussion about a limited set of sources

Trade-offs

  • It is less suited than Researcher for comprehensive, multistep research.
  • The resulting response may be less complete or less report-oriented.
  • The team may need to coordinate and structure the research manually.

Alternative: Use Analyst in Microsoft 365 Copilot

Analyst is intended for deeper data analysis, including working with structured data, calculations, and analytical models.

Why select this alternative

It is appropriate when the task requires:

  • Statistical analysis
  • Spreadsheet or tabular data exploration
  • Quantitative comparisons
  • Calculations, trends, or forecasting
  • Code-assisted analysis

Trade-offs

  • It is unnecessary when the primary task is qualitative research and synthesis.
  • The scenario does not require deep statistical modeling or code.
  • Researcher is a better initial experience for the briefing described.

Alternative: Start Directly in Copilot for PowerPoint

Copilot in PowerPoint can create or revise a presentation from a prompt or source document.

Why select this alternative

It is useful after the research and narrative have been established.

Trade-offs

  • PowerPoint Copilot is primarily a presentation-authoring experience.
  • It should not be the first choice for deep, multi-source research.
  • Starting with slides too early can produce a polished presentation without sufficiently defensible evidence.

Using Microsoft 365 Copilot Chat as a Unified, Cross-Device Work Assistant

Problem / Scenario / Need

Organizations want employees to have one consistent, secure AI chat experience across desktop, web, and mobile rather than a patchwork of different tools per device. In practice this raises several related questions:

  • Is it accurate to say Microsoft 365 Copilot Chat provides a genuinely unified experience across the web and the mobile app?
  • How is Copilot Chat actually different from Microsoft 365 Search, for a user who assumes Copilot Chat is "just another search box"?
  • What can a mobile user do that a desktop user cannot — for example, turning a photographed paper note into usable digital text?
  • Which app should a commercial user sign in to, given that Microsoft also ships a separate consumer-facing Copilot app?

The solution should let users continue conversations across devices, work with permitted Microsoft 365 files, use voice or text away from a desk, capture information from the physical world into digital text, and remain inside the organization's Microsoft 365 identity, security, and compliance boundary.

Solutions

Best Practice: Use Microsoft 365 Copilot Chat via the Microsoft 365 Copilot App as the Single, Permission-Aware Assistant Across Surfaces

The best fit for a consistent, cross-device AI assistant is Microsoft 365 Copilot Chat, accessed through the Microsoft 365 Copilot app (available on web, desktop, iOS, and Android) or embedded directly in Microsoft 365 apps.

Microsoft 365 Copilot Chat is a conversational AI assistant, not a search engine. For a user signed in with a Microsoft Entra ID work or school account, it can reason over permitted organizational content — files, email, meetings, chats — together with web information, to:

  • Answer questions and explain topics
  • Summarize files, meetings, and communications
  • Find and synthesize information across multiple work sources
  • Draft emails, briefs, agendas, and presentation content
  • Brainstorm and iteratively refine content
  • Save reusable output as a Copilot Page

Because the same app and identity are used everywhere, the experience genuinely travels with the user: a conversation started on a laptop can be picked up on a phone, files remain scoped to what the user could already access, and the organization's security and compliance controls continue to apply regardless of device.

Why this is the best practice

  • Versus Search: Search only locates existing items. Copilot Chat goes further by synthesizing, comparing, and generating new work product from the same permitted content, and it keeps conversational context for follow-up questions — that is the concrete answer for a user who assumes Copilot Chat is "just another search box."
  • Versus fragmenting the experience by device: a single app and identity across web, desktop, iOS, and Android means users learn one interface, IT manages one access/security surface, and conversation history genuinely carries across devices — instead of employees having to reconcile two disconnected tools.
  • Permission-aware by design: a Copilot license or Copilot Chat access never grants a user access to content they could not already open themselves.
  • Scales by license tier without fragmenting the app: users without a paid Microsoft 365 Copilot add-on still get a secure, web-grounded chat experience through the same app and account; users with the add-on additionally get the work-grounded experience (reasoning over their own files, email, meetings, and chats).

Distinguishing the Commercial App from the Consumer App

Microsoft ships two different apps that are easy to confuse:

  • Microsoft 365 Copilot app — the commercial/education productivity app, used with a Microsoft Entra ID work or school account. This is the app that applies the organization's security, compliance, and data-handling controls.
  • Microsoft Copilot app — a separate, consumer-oriented app for personal Microsoft accounts, without the organization's commercial data protection.

This distinction matters beyond naming: if an employee signs in to the consumer app with work content, or defaults to it out of habit, that conversation falls outside the organization's compliance boundary. Commercial users should be directed specifically to the Microsoft 365 Copilot app.

Capturing Physical Notes on Mobile with Scan Text

Field or on-site workers can turn paper notes, checklists, or whiteboards into usable digital text directly from the Microsoft 365 Copilot mobile app, using the camera-based Scan Text action:

  1. Open the Microsoft 365 Copilot mobile app.
  2. Select the create/add action, then open Documents.
  3. Select Scan Text.
  4. Photograph the paper notes, checklist, or whiteboard.
  5. Review and save the converted (OCR'd) text.
  6. Use the resulting text as context in Copilot Chat to summarize it, draft an email, or create work-item content.

This closes the gap between paper-based fieldwork and Microsoft 365 content: it avoids manual retyping, and the extracted text becomes immediately reusable in Copilot and other Microsoft 365 applications.

Important Qualification: Exact Capabilities Depend on Licensing, Permissions, and Platform

  • Web-grounded Copilot Chat is available broadly to users with a qualifying Microsoft Entra ID work or school account in an eligible tenant, without requiring the paid Microsoft 365 Copilot add-on.
  • The fuller, work-grounded experience — reasoning over the user's own files, email, meetings, and chats — requires the user to be assigned a Microsoft 365 Copilot add-on license.
  • Administrator configuration, the user's existing permissions to work content, mobile OS/app version, regional or organizational rollout, and per-feature mobile support can all affect which capabilities (including Scan Text and specific agents) are actually available to a given user.
  • None of this expands what the user could already access — Copilot Chat only reasons over content the signed-in user already has permission to open.

Alternative: Use Microsoft 365 Search Instead of Copilot Chat

Microsoft 365 Search is appropriate when the user's actual need is to locate a specific, known item: a file, person, meeting, chat, or recently shared item.

Trade-offs

  • Search returns existing resources; it does not summarize, compare, reason over, or transform them into a new work product.
  • It does not hold conversational, multi-turn context the way Copilot Chat does.
  • For anything beyond "find this item," Copilot Chat is the more capable tool.

Alternative: Use Scan to PDF Instead of Scan Text

Why select this alternative

Scan to PDF (in the mobile app or OneDrive) is preferable when preserving the original layout matters, multiple pages must be captured together, or the image itself needs to stay available for visual review — for example, an annotated diagram or a multi-page paper form.

Trade-offs

  • The captured content is less directly editable than the plain text Scan Text produces.
  • Handwriting and low-quality images still limit OCR accuracy if Copilot is asked to read the PDF's contents.
  • Extracted details should be verified before they are used to create operational records.

Alternative: Upload a Normal Photo Instead of Using Scan Text

Why select this alternative

A standard photo attached directly to Copilot Chat is faster for an informal, one-off question where no dedicated document is needed.

Trade-offs

  • The image may include glare, distortion, or irrelevant background detail that a scanning mode would crop or clean up.
  • OCR-oriented modes like Scan Text generally produce cleaner, more directly reusable text.
  • Support depends on the specific app, account, and current feature availability.

Alternative: Build a Custom Agent with Microsoft Copilot Studio

Why select this alternative

Appropriate when the assistant must follow organization-specific workflows, ground itself in custom knowledge sources beyond Microsoft 365, call business systems or APIs, or take defined actions — needs that go beyond general knowledge-worker productivity chat.

Trade-offs

  • It is unnecessary overhead when the actual need is general-purpose productivity assistance, which Copilot Chat already covers.
  • The organization must still decide how the custom agent is surfaced consistently across devices; it does not automatically inherit the Microsoft 365 Copilot app's cross-device presence.

Why select this alternative

May be necessary when a specialized field workflow genuinely requires a dedicated mobile application with capabilities the Microsoft 365 Copilot app does not provide.

Trade-offs

  • Users must learn and switch between different interfaces.
  • Conversation and context continuity across devices is lost.
  • Governance, licensing, and support responsibilities roughly double.

Choosing and Sequencing Researcher and Analyst in Microsoft 365 Copilot

Problem / Scenario / Need

Microsoft 365 Copilot includes two specialized reasoning agents — Researcher and Analyst — and teams need to get two things right:

  1. Which agent fits a given task (a common exam trap swaps their roles).
  2. How to combine them when a single deliverable needs both a source-backed narrative and a data-driven quantitative analysis — for example, a strategic briefing, an executive board package, or a go/no-go dossier that must synthesize market and competitor evidence and validate it against internal sales, pipeline, or performance datasets.

Solutions

Best Practice: Match Each Task to the Right Agent, and Run Researcher Before Analyst When a Deliverable Needs Both

  • Researcher is built for complex, multistep research. It combines the web with permitted work content (files, email, meetings, chats) and produces a structured, source-cited report or narrative. It is the right tool whenever the deliverable is primarily qualitative synthesis across many sources — a market-entry report, a competitor comparison, a regulatory summary, or a go/no-go dossier.
  • Analyst is built for data-heavy analysis. It works like an on-demand data scientist over structured data (spreadsheets, CSVs, tables), running code-backed (Python) analysis behind the scenes to calculate, detect trends and outliers, compare periods, answer interactive "what-if" questions, and generate charts. It is the right tool whenever the deliverable is primarily quantitative investigation of a dataset the team already has.

A frequent exam trap reverses these roles (claiming Researcher is for tables/code/charts and Analyst is for multi-source cited reports) — the roles are exactly the opposite of that claim.

When a single deliverable needs both a narrative and a data-backed validation — such as a strategic briefing or board package — the efficient sequence is:

  1. Researcher establishes the strategic narrative and evidence base, and — critically — identifies which internal metrics and questions matter enough to analyze.
  2. Analyst examines the specific internal datasets Researcher's questions point to, and produces calculations, trend/outlier findings, and a small number of decision-relevant charts.
  3. The team combines both outputs (in Word, PowerPoint, or Copilot Chat) into a single package, keeping external evidence/citations, internal calculations/reporting periods, and management interpretation clearly distinguished, and resolving any inconsistencies between the two outputs before executive use.

Example prompts:

Research the major market, competitor, and regulatory developments affecting our business. Use current web sources and our permitted internal strategy content. Produce a sourced executive briefing, identify the five issues most relevant to the board, and specify which internal performance measures should be analyzed.

Analyze the attached revenue, margin, customer-retention, and regional-performance data in relation to these five issues from the Researcher report. Identify material trends and anomalies, create three board-ready charts, and explain the decision implications.

Create a concise board briefing using the Researcher report and Analyst findings. Include an executive summary, three charts, key risks, recommended decisions, and five speaking points. Preserve the research citations and state the source and reporting period for every internal metric.

Reverse the sequence — Analyst first, then Researcher — when an unexpected internal result is what triggers the work, for example a sudden margin decline, a sharp increase in churn, or an unexplained regional variance. In that case Analyst quantifies the internal anomaly first, and Researcher then investigates external (market, competitor, regulatory) explanations for it.

Why this is the best practice

Picking the wrong agent for a single-purpose task wastes the exercise: Researcher is not optimized to run precise calculations, detect statistical outliers, or produce charts from a raw dataset, and Analyst is not built to gather or cite broad qualitative evidence from the web and work content. For a task that only needs one kind of reasoning, using the matching agent directly is faster and more reliable than trying to force the other agent to do it.

For a mixed deliverable, sequencing matters because the two agents answer different questions: Researcher answers "what does the broader evidence say, and which internal numbers are worth checking?"; Analyst answers "what do those specific numbers actually show?" Starting with Researcher gives Analyst a target instead of an open-ended, unfocused data exploration; running Analyst first only makes sense when an internal number is already the anomaly that needs an external explanation. Either way, the final synthesis step is what prevents the narrative and the data from remaining two disconnected documents that executives have to reconcile themselves.

Quick Reference: Which Agent for Which Task

Use Researcher for:

  • Preparing a market-entry or go/no-go report
  • Comparing competitors using internal and public information
  • Researching a regulatory topic
  • Synthesizing emails, meetings, files, and web sources into one narrative
  • Producing a detailed report with citations

Use Analyst for:

  • Comparing sales performance across regions
  • Finding unusual values in a large operational dataset (e.g., warehouse picking-error events)
  • Exploring customer churn data
  • Comparing budget vs. actual figures and plotting quarterly trends
  • Letting a frontline manager self-serve trend/outlier analysis without waiting on the BI team
  • Calculating trends, correlations, and creating charts from uploaded tables

Alternative: Use Only One Agent for a Mixed Narrative-and-Data Task

Rely on Researcher or Analyst alone even though the deliverable has both narrative and quantitative components.

Why select this alternative

  • Researcher alone can be sufficient when the "quantitative" component is limited to figures already reported inside documents, and the team does not need to inspect raw or structured datasets directly.
  • Analyst alone can be sufficient when the entire assignment is quantitative, all relevant information already exists in structured datasets, and no external market/competitor research is required.

Trade-offs

  • Researcher alone is not the strongest tool for detailed dataset exploration: outlier detection, exact calculations, and quarter-over-quarter comparisons are less rigorous than Analyst would produce.
  • Analyst alone will not produce the same depth of cited, multi-source market synthesis that Researcher is designed for — external reports and internal narrative documents still need a research-oriented workflow.
  • Using one agent to approximate the other's job typically means redoing part of the work once the gap is noticed.

Alternative: Use Copilot Chat for the Entire Workflow

Copilot Chat can summarize files, answer questions, and support iterative drafting without invoking either specialized agent.

Why select this alternative

Reasonable for a quick answer, brainstorming, or a small, well-bounded set of sources where neither extended multi-step research nor dataset-level computation is really needed.

Trade-offs

  • Copilot Chat does not run the extended, multi-step research loop Researcher uses to gather and cite many sources, so citation depth and coverage will be thinner.
  • Copilot Chat does not execute code the way Analyst does, so it cannot reliably compute exact statistics, detect outliers in a large dataset, or generate charts from raw data.
  • The user ends up manually coordinating and stitching together work that Researcher and Analyst would otherwise structure automatically.
